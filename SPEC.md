# 論文和訳PDF書き込みツール — 詳細指示書(仕様書)

**版**: Draft v0.2(2026-07-18)
**対象読者**: 本ツールの実装者
**利用者想定**: 化学系の研究者・学生(有機化学・無機化学・分析化学・物理化学・高分子・材料化学など)

本書は実装のための指示書であり、この段階ではコードの実装は行わない。
既存の草案コード(main.py / pdf_translator.py / translators/)は本書に従って改修する。

---

## 1. 目的と全体像

英語の化学系学術論文PDFを入力とし、本文テキストを **正確な化学用語の和訳** で置換した
PDFを出力する。レイアウト(2段組・図表・数式の位置)は極力維持する。

```
入力PDF → ①抽出 → ②ブロック分類 → ③化学用語保護 → ④翻訳 → ⑤検証 → ⑥PDF書き込み → 出力PDF
```

## 2. 機能要件

### 2.1 入力
- テキスト選択可能なデジタルPDF(英語)。スキャンPDFは対象外(エラーメッセージで明示すること)
- 暗号化PDFはパスワード指定オプション `--password` で対応
- 想定ジャーナル形式: ACS系(JACS, Org. Lett. 等)、RSC系、Elsevier系、Wiley系、Springer系の2段組・1段組

### 2.2 出力
- 原文を白塗り消去し、同位置に和訳を描画したPDF(既定: `<入力名>_ja.pdf`)
- `--mode sidecar` オプション: 原文PDFはそのまま、各ページの直後に和訳ページを挿入する対訳モード(将来拡張、優先度低)
- 処理レポート(`<出力名>.report.json`): 翻訳したブロック数、スキップしたブロックとその理由、未収録用語の一覧、API使用量概算

### 2.3 CLI
```
python main.py --input paper.pdf [--output paper_ja.pdf]
               [--engine claude|deepl|google]
               [--glossary my_glossary.csv]      # 追加ユーザー用語集
               [--field organic|inorganic|analytical|physical|polymer|materials|general]
               [--pages 1-5,8]                   # 対象ページ指定
               [--dry-run]                       # 抽出・分類結果の確認のみ
               [--review-tsv out.tsv]            # 対訳TSVを併せて出力(人手検証用)
```

## 3. ブロック分類仕様(②)

抽出した各テキストブロックを以下に分類し、**翻訳対象/保護対象/除外** を決める。

| 分類 | 判定基準(例) | 扱い |
|---|---|---|
| 本文段落 | 文字数 ≥ 40、文末ピリオドあり | 翻訳 |
| 見出し | 短文+フォントサイズが本文より大/太字、"Introduction" 等の既知見出し | 翻訳(定訳表 §5.4 を使用) |
| 図表キャプション | "Figure N", "Fig. N", "Table N", "Scheme N", "Chart N" で始まる | 翻訳(ラベル部 "Figure 1." は「図1.」等に変換) |
| 数式・化学式ブロック | 数式フォント(CMMI等)率が高い、または §4 の化学式パターンに全体が一致 | **除外(そのまま残す)** |
| 参考文献 | "References" 見出し以降、番号+著者名パターン | **除外(既定)**。`--translate-refs` で翻訳可 |
| ヘッダ/フッタ/ページ番号 | ページ上下端の短小ブロック、全ページ反復 | 除外 |
| 著者・所属・メールアドレス | 1ページ目上部、@を含む等 | 除外 |
| 脚注・謝辞・利益相反 | "Acknowledg", "Conflict of interest", "Supporting Information" 等 | 翻訳 |

判定に迷うブロックは「翻訳」に倒し、report.json に `classification: "uncertain"` として記録する。

## 4. 化学用語・表記の保護仕様(③)— 本ツールの中核

翻訳エンジンに渡す **前** に、以下のパターンをプレースホルダ(`⟦CHEM_001⟧` 形式)に
置換して保護し、翻訳 **後** に復元する。翻訳エンジンによる化学式・単位の破壊を防ぐ。

### 4.1 無変換で保護するもの(プレースホルダ化)
1. **化学式・組成式**: `H2O`, `C6H5OH`, `[Ru(bpy)3]2+`, `Fe2O3`, `CH3CH2OH` など
   - 判定: 元素記号の連続+数字/括弧/電荷表記の正規表現。上付き・下付きのスパン情報も利用
2. **SMILES / InChI 文字列**: `InChI=` で始まる文字列、SMILES らしき記号列
3. **化合物番号**: 太字の `1`, `2a`, `(3b)` 等(化学論文特有の化合物ラベル)。太字スパン+英数字短パターンで検出
4. **単位付き数値**: `25 °C`, `1.5 mmol`, `300 MHz`, `δ 7.26 ppm`, `λmax = 450 nm`, `Rf = 0.3`, `mp 120–122 °C` など
   - 単位辞書: mol, mmol, μmol, M, mM, g, mg, μg, mL, μL, °C, K, ppm, MHz, Hz, nm, cm⁻¹, eV, kcal/mol, kJ/mol, Å, mV, mA/cm² 等
5. **NMR/MS/IR データ列**: `1H NMR (400 MHz, CDCl3): δ 7.26 (s, 1H), ...` のようなスペクトルデータ行は **行全体を保護**(翻訳しない)
   - 判定: `NMR`, `HRMS`, `ESI-MS`, `IR (KBr)`, `Anal. Calcd` 等の開始パターン
6. **装置名・試薬グレード・型番**: `Bruker Avance III`, `Sigma-Aldrich`, `JASCO V-670` 等の固有名詞(NERは行わず、大文字始まり連続語+既知メーカー辞書で近似)
7. **遺伝子/タンパク質表記・ギリシャ文字混じり記号**: `α,β-unsaturated` の `α,β-` 部分等の位置指定子(§4.2 の定訳と連携)
8. **DOI・URL・データベースID**: `10.1021/...`, CCDC番号, CAS RN

### 4.2 定訳辞書で強制置換するもの(用語集)
翻訳エンジン任せにせず、**内蔵用語集**で訳語を固定する。実装は「保護→翻訳→復元」時に
訳語をプレースホルダの復元値として与える方式、または LLM系エンジンではプロンプトの
用語集セクションに列挙する方式(§6.1)の **両方** を用意し、エンジンごとに使い分ける。

内蔵用語集は `glossaries/` ディレクトリに TSV で同梱する:

```
glossaries/
├── core_chemistry.tsv      # 全分野共通(必ず読み込む)
├── organic.tsv             # --field organic で追加読み込み
├── inorganic.tsv
├── analytical.tsv
├── physical.tsv
├── polymer.tsv
└── materials.tsv
```

TSV形式: `英語⇥日本語訳⇥備考(任意)`。マッチは最長一致・大文字小文字無視(ただし略語は完全一致)。

**core_chemistry.tsv に収録すべき用語の例**(実装時に300語以上へ拡充すること):

| 英語 | 定訳 | 備考 |

| moiety | 部分構造 / 〜部位 | |
| substituent | 置換基 | |

| catalyst / catalytic | 触媒 / 触媒的 | |
| stoichiometric | 化学量論量の | |
| reflux | 還流 | |
| workup | 後処理 | 「ワークアップ」も許容 |
| quench | クエンチする(反応停止) | |

| recrystallization | 再結晶 | |
| enantioselectivity / ee | エナンチオ選択性 / ee | ee は保護 |
| diastereomer | ジアステレオマー | |
| regioselectivity | 位置選択性 | |
| oxidative addition | 酸化的付加 | |
| reductive elimination | 還元的脱離 | |
| transmetalation | トランスメタル化 | |

### 4.3 ユーザー用語集
`--glossary` で追加TSVを読み込み、内蔵用語集より **高優先** で適用する。
研究室固有の訳語運用(例: "complex" を「錯体」に固定)を可能にする。

## 5. 翻訳仕様(④)

### 5.1 文体
- 学術文体・である調
- 受動態の英文は日本語として自然なら能動に転換してよい
- 数値・有効数字・不等号・±表記は改変禁止

### 5.2 文脈供給
- ブロック単独ではなく、**同一ページの前後ブロックの原文**を文脈として翻訳エンジンに供給する(LLM系のみ)
- 論文タイトルとアブストラクトを最初に翻訳し、以降の全リクエストに「この論文の主題」として添付する(訳語の一貫性向上)

### 5.3 訳語一貫性
- 一度訳した専門用語は文書全体で同一訳語を使う。実装: 翻訳済みペアを蓄積する **動的用語集**(セッション内キャッシュ)を作り、以降のプロンプト/用語集に自動追加
- report.json に「本文書で使用した訳語対応表」を出力する

### 5.4 定型見出しの定訳表
| 英語 | 訳 |
|---|---|
| Abstract | 要旨 |
| Introduction | 緒言 |
| Results and Discussion | 結果と考察 |
| Experimental Section / Materials and Methods | 実験 / 実験方法 |
| General Procedure | 一般的手順 |
| Conclusion(s) | 結論 |
| Acknowledgment(s) | 謝辞 |
| Supporting Information | サポート情報 |
| References | 参考文献(既定では翻訳せず残す) |

## 6. 翻訳エンジン別仕様

### 6.1 Claude(既定・推奨)
- モデル: `claude-sonnet-5`(config.yaml で変更可)
- システムプロンプトに以下を含める:
  1. 「化学系学術論文の専門翻訳者」であるという役割定義
  2. §5.1 の文体規則
  3. `--field` に応じた分野宣言(例: 「本論文は有機合成化学の論文である」)
  4. 適用用語集(該当ブロックに出現する用語のみ抽出して列挙。全用語を毎回送らない — トークン節約)
  5. プレースホルダ `⟦CHEM_nnn⟧` は **絶対に改変・削除・翻訳しない** こと
  6. 入出力はJSON配列、要素数厳守
- 温度は低め(temperature=0.2 相当)を指定
- 出力検証: 要素数一致 + 全プレースホルダの生存確認。失敗時は当該バッチをブロック単位に分割して再試行

### 6.2 DeepL
- 用語集は DeepL Glossary API に事前登録して glossary_id を指定する方式
- `tag_handling="xml"` を使い、プレースホルダを `<x id="nnn"/>` タグとして保護
- 文脈供給(§5.2)は `context` パラメータで論文タイトル+要旨を渡す

### 6.3 Google Cloud Translation
- Advanced (v3) の Glossary 機能を使用
- プレースホルダ保護は自前実装(§4.1 と同じ)
- 化学用語精度は3エンジン中最も落ちるため、README に「化学論文には claude / deepl を推奨」と明記

## 7. 翻訳後検証(⑤)

書き込み前に自動チェックし、失敗ブロックは report.json に記録して **原文のまま残す**(壊れた訳を書き込むより安全):

1. プレースホルダ全数復元チェック
2. 数値検証: 原文中の数値(小数・指数表記含む)が訳文にすべて存在するか
3. 空訳・原文コピー検出(訳文が空、または原文と同一)
4. 長さ異常検出: 訳文が原文の3倍超/0.2倍未満なら要注意フラグ
5. `--review-tsv` 指定時: `ページ⇥ブロックID⇥原文⇥訳文⇥フラグ` のTSVを出力し、人手レビューを可能にする

## 8. PDF書き込み仕様(⑥)

- 草案どおり redaction(白塗り)+ `insert_textbox` 方式を維持
- フォント: Noto Sans JP を埋め込み。**上付き・下付きは再現しない**(保護されたスパンは原文画像がそのまま残るため実害は限定的だが、制約として README に明記)
- はみ出し処理: フォント縮小(下限 6pt)→ それでも収まらない場合は矩形を下方向に 10% まで拡張 → なお不可なら原文のまま残し report.json に記録
- ギリシャ文字・±・°・Å 等が描画可能なことをフォント読み込み時に検証する

## 9. 非機能要件

- **レート制限**: エンジンごとに RPM 上限を config.yaml で設定し、超過時は指数バックオフ(最大3回)
- **コスト表示**: 実行前に総文字数からAPI費用概算を表示し、`--yes` がなければ確認プロンプトを出す
- **中断再開**: ブロック単位の翻訳結果を `.cache/<入力PDFのハッシュ>.json` にキャッシュし、再実行時は未翻訳ブロックのみAPIに投げる
- **ログ**: `--verbose` で分類・保護・翻訳の各段階をログ出力
- **配布**: APIキーは `.env` のみ。用語集TSVは同梱し、利用者が編集できることを README に記載

## 10. 論文要約機能

翻訳と同じ抽出結果(①〜②)を再利用し、論文全体の日本語要約を生成する。

### 10.1 要件
- **対応エンジン**: LLM系(Claude)のみ。DeepL/Google選択時に要約を指定した場合、要約のみClaudeを使う(Claudeキー未設定ならエラーで案内)
- **入力**: 分類済みブロックのうち本文・見出し・キャプション(参考文献・ヘッダ等は除外)。長編は見出し構造で分割し、セクション要約→統合要約の2段階(map-reduce)で処理
- **化学系テンプレート**(既定): 以下の構造化要約を生成する
  1. **研究目的・背景**(2〜3文)
  2. **手法**: 合成経路・測定手法・計算手法の要点
  3. **主要な結果**: 収率・選択性・物性値など**原文の数値を保持**して列挙(§7の数値検証を要約にも適用)
  4. **結論と意義**(2〜3文)
  5. **キーワード**(5〜8語、用語集の定訳を使用)
- 用語集(§4.2〜4.3)は要約プロンプトにも適用し、翻訳本文と訳語を一致させる
- 長さ指定: `short`(約200字)/ `standard`(約600字、既定)/ `detailed`(約1500字)

### 10.2 出力形式(複数選択可)
- **PDFへの挿入**(既定): 出力PDFの先頭に「要約」ページを1枚追加(和訳本文と同じ日本語フォント)
- **別ファイル**: `<出力名>.summary.md`
- report.json にも `summary` フィールドとして格納

### 10.3 CLI
```
--summary                  # 要約を生成(既定: standard、PDF先頭に挿入)
--summary-length short|standard|detailed
--summary-only             # 翻訳せず要約だけ生成(高速・低コスト用途)
```
`--summary-only` は本文翻訳を行わないため、Readable的な「まず要約で読む価値を判断→必要なら全文翻訳」という使い方を想定する。要約結果は翻訳と同様にキャッシュする(§9)。

## 11. 実装順序(マイルストーン)

1. **M1**: ブロック分類(§3)+ dry-run での分類結果可視化
2. **M2**: 化学用語保護・復元(§4.1)+ 用語集読み込み(§4.2, 4.3)
3. **M3**: Claude エンジンの新プロンプト(§6.1)+ 翻訳後検証(§7)
4. **M4**: 書き込み改善(§8)+ report.json / review-tsv 出力
5. **M5**: DeepL / Google の用語集対応(§6.2, 6.3)、キャッシュ・コスト表示(§9)
6. **M6**: 要約機能(§10)

## 12. テスト計画(概要)

- ACS / RSC / Elsevier 各1報のオープンアクセス論文でE2E確認
- 保護パターンの単体テスト: 化学式・NMRデータ行・単位付き数値の正規表現に対し、正例/負例各20件以上
- 用語集適用テスト: core_chemistry.tsv の全用語が訳文に反映されることを機械検証
- 検証ロジックテスト: プレースホルダ欠落・数値欠落を意図的に起こして検出できること
